## Critical

### Header 'Tool Permission Level' control is a decoy — approval level it sets is never enforced

_Surface: `chat-agent-interaction`_

The lock-icon approval dropdown (both wide header and narrow overflow menu) writes approvalLevel into CHANNEL config via updateChannelConfig, but no agent code reads channel-config approvalLevel: the agent loop's approval gate reads state.config.approvalLevel, which loopConfig() builds solely from per-agent stored settings (seeded from subscription config or the setApprovalLevel method). The vessel's only channel-config read (getCachedChannelConfig) is used exclusively for conversationPolicy/agentHopLimit in shouldRespond. Since the per-agent default is 2 (Full Auto), a user who selects 'Ask All' believing they've locked the agent down continues to get unprompted writes, evals, and web calls — a consent control that silently does nothing, with data-destruction potential and no error. The header also displays this dead channel value ('Full Auto' when unset) even when the real per-agent level was changed via the agent Settings dialog, so the two approval controls in the same UI can disagree.

**Evidence:** workspace/packages/tool-ui/src/hooks/useToolApproval.ts:85-96 (writes updateChannelConfig({approvalLevel})); workspace/packages/agentic-chat/components/ChatHeader.tsx:271-276,411-430; workspace/packages/agent-loop/src/policies/index.ts:112 (gate reads state.config.approvalLevel); workspace/packages/agentic-do/src/agent-vessel.ts:1242 (loopConfig approvalLevel from getAgentSettings only), 1974+2138 (getCachedChannelConfig used only in shouldRespond); grep of agentic-do shows no channel-config→approvalLevel bridge; real setter is AgentDialog.tsx:197 (setApprovalLevel method); default level 2: workspace/packages/agentic-do/src/agent-config.ts:21

**Recommended fix:** Either make the header control call setApprovalLevel on every agent participant (and read the displayed value from getAgentSettings), or make the vessel fold channel-config approvalLevel into its loop config; remove whichever path is not chosen so one authoritative control remains.

## High

### Unloadable workspace config still exits the app silently before any window exists

_Surface: `startup-onboarding`_

resolveStartupMode() runs at module load, before any window is created. If the last-opened workspace's meta/vibestudio.yml is present but unparseable (invalid YAML — plausible since workspace agents and users edit this file), parseWorkspaceConfigContent throws, and the catch does console.error + app.quit() + process.exit(1). Because dialog.showErrorBox is monkeypatched to a console logger, a GUI user double-clicks the app and literally nothing appears — no window, no dialog, no recovery path. This is the remaining unfixed slice of the prior 'silent startup exit' critical: failures after the bootstrap window exists now land on a recovery screen, but failures during startup-mode resolution do not.

**Evidence:** src/main/index.ts:191-197 (catch → app.quit()/process.exit(1), console only); src/main/index.ts:60-73 (dialog.showErrorBox monkeypatched to console); packages/workspace/src/loader.ts:468-480 (loadWorkspaceConfig throws on missing/corrupt vibestudio.yml); packages/workspace/src/startup.ts:74-81 (last-opened path resolves without init and rethrows)

**Recommended fix:** On resolveStartupMode failure, fall back to startupMode = { kind: "pending" } and set bootstrapStartupError with the parse error message so the existing bootstrap recovery/chooser window opens and shows the reason, instead of exiting.

### Launching a workspace whose config file is missing silently deletes the entire workspace directory

_Surface: `startup-onboarding`_

resolveOrCreateWorkspace with init:true treats any existing wsDir lacking source/meta/vibestudio.yml as an 'interrupted create' and does fs.rmSync(wsDir, { recursive: true, force: true }) before re-templating — deleting source/panels, projects, and all state without confirmation. Every workspace-switch relaunch appends --workspace-create-if-missing (init:true), and the bootstrap chooser's 'Launch by name'/'Create and launch' path also passes init:true. So a workspace whose manifest was deleted or misplaced (agent mishap, partial git operation, version skew) but which still contains real user work is wiped wholesale the moment the user tries to open it.

**Evidence:** packages/workspace/src/loader.ts:593-604 (existsSync(configPath) false + init → rmSync(wsDir, recursive)); src/main/startupMode.ts:146-153 (workspaceRelaunchArgs always appends WORKSPACE_CREATE_IF_MISSING_ARG); src/main/index.ts:1894-1903 (chooser local choice passes init: true)

**Recommended fix:** Only rm-and-recreate when the directory matches a genuine partial create (e.g., a sentinel/marker written during initWorkspace, or an empty/near-empty dir). Otherwise fail with an actionable error ('workspace exists but has no manifest — restore meta/vibestudio.yml or choose a different name') and never delete user data implicitly.

### Piped CLI output larger than the 64KB pipe buffer is silently truncated with exit 0

_Surface: `cli-interactive`_

The CLI entry point calls process.exit(code) as soon as main() resolves. On Linux, writes to a pipe are asynchronous, so any output still queued in the stdout stream is dropped when the process exits. `vibestudio fs read bigfile > local.bin` (binary-safe read is a headline feature of fs read) or any large --json result piped to jq will be cut off at ~64KB while the command still reports success (exit 0). Verified the mechanism: `node -e 'process.stdout.write(Buffer.alloc(8*1024*1024,65)); process.exit(0)' | wc -c` yields 65536, and the built dist/cli/client.mjs ends with the same .then(t=>process.exit(t)) pattern. Silent data corruption with a success exit code and no error is the worst failure shape for a scripting surface.

**Evidence:** src/cli/client.ts:1195-1202 (main(...).then((code) => process.exit(code))); src/cli/agent/fsCommands.ts:100 (process.stdout.write(buf) for fs read); src/cli/output.ts:96 (console.log(JSON.stringify(value)) for JSON results); dist/cli/client.mjs tail shows identical exit pattern; empirical pipe test truncated 8MB to 65536 bytes

**Recommended fix:** Replace process.exit(code) with process.exitCode = code and let the event loop drain naturally (close any lingering clients so the loop can empty), or explicitly await a stdout 'drain'/write callback before exiting.

### vibestudio remote setup-signaling is unusable from npm installs and fails with a cryptic spawn ENOENT

_Surface: `cli-packaging-scripts`_

The command resolves wrangler at <packageRoot>/node_modules/.bin/wrangler and cd's into <packageRoot>/apps/signaling, but wrangler is only a repo devDependency (not in computeHostDependencies) and apps/ is never staged into either published package (files list is dist/vendor/workspace-template/patches/scripts). A packaged user who follows docs/cli.md's 'Use remote setup-signaling ... to self-host the signaling worker' gets '[remote-setup-signaling] spawn .../node_modules/.bin/wrangler ENOENT' with no hint that a source checkout is required.

**Evidence:** scripts/cli/remote-setup-signaling.mjs:9-11,85; scripts/build-npm-packages.mjs:173,232 (no apps/, no wrangler dep); docs/cli.md:140-143

**Recommended fix:** Detect the packaged layout (fs.existsSync(signalingDir) and the wrangler bin) up front and fail with 'remote setup-signaling requires a vibestudio source checkout (git clone ... && pnpm bootstrap)', or ship the worker source and depend on wrangler; also mention the source-checkout requirement in docs/cli.md.

### Closing a panel silently archives its entire subtree with no confirmation and no undo

_Surface: `shell-chrome-keyboard`_

Every close affordance — the hover X on breadcrumbs, the hover X in the tree sidebar, the context-menu Archive item, and the Close Panel accelerator (Cmd+W / Ctrl+Shift+W) — immediately archives the panel AND all descendant panels ('Server authority closes the subtree'). There is no confirmation even when the panel has many children, no undo toast, and no unarchive/restore surface anywhere in the shell UI (no 'unarchive' consumer exists in workspace/apps/shell or workspace/about). One accidental click on a 16px hover button can take down a whole working tree of panels with no recourse.

**Evidence:** workspace/apps/shell/components/TitleBar.tsx:1037-1047 (breadcrumb X calls panel.archive directly), workspace/apps/shell/components/LazyPanelTreeSidebar.tsx:602-618 (tree row X), src/main/menu.ts:406-411 and 510-516 (Close Panel accelerator -> archiveFocusedPanel), src/main/panelOrchestrator.ts:521-527 (archive closes the subtree), grep for 'unarchive' in shell UI returns nothing

**Recommended fix:** Confirm before archiving a panel that has descendants (e.g. 'Close "X" and its 4 child panels?'), and/or show an undo toast after archive backed by a restore call; expose an 'archived panels' list to recover from.

### Panel connection-error infrastructure is dead code — no host ever emits runtime:connection-error

_Surface: `panels-infra-notifications`_

The panel SDK ships a full connection-loss UX: panel.onConnectionError (createBaseRuntime.ts) and the ConnectionErrorBarrier in the React mount that renders a full-screen 'Connection lost' overlay or a 'Backend unavailable' banner. PANEL_SYSTEM.md documents it as firing 'on terminal WebSocket close codes'. But a repo-wide search finds zero emitters of the 'runtime:connection-error' event — not in ipcDispatcher, wsTransport, panelOrchestrator, or the mobile host. When the panel's RPC session dies (lease revoked, server down, pipe torn down), the overlay never appears; and since the RPC client has no implicit request deadline, panel calls simply hang — the panel looks frozen with no on-screen explanation.

**Evidence:** workspace/packages/runtime/src/setup/createBaseRuntime.ts:180-198 (only listener); workspace/packages/react/src/reactPanel.ts:83-147 (overlay UI); PANEL_SYSTEM.md:259-274 (docs); grep for 'runtime:connection-error' across src/, workspace/, packages/ returns only the listener; packages/rpc/src/client.ts:508-510 ('No implicit deadline')

**Recommended fix:** Emit runtime:connection-error from the desktop host when the panel session terminally closes or the server pipe enters a terminal state (ipcDispatcher.ensurePanelSession catch / session close callback, panelOrchestrator lease-revoked path), and from the mobile bridge equivalently; add a reconnect/retry affordance to the ConnectionErrorBarrier overlay.

### Sensitive permission requests from panels are silently denied with no consent path

_Surface: `panels-infra-notifications`_

The default-deny permission handler rejects geolocation, media (camera/mic), fullscreen, pointerLock, display-capture, notifications, etc. for every panel and browser-panel webContents — only app views with a matching manifest capability can pass. The denial is console.warn-only; there is no consent prompt, no NotificationBar surfacing, and no settings surface to grant. Concretely: clicking fullscreen on a video in a browser panel does nothing, and any site needing camera/mic just fails, with no explanation anywhere on screen and no recourse.

**Evidence:** src/main/index.ts:1689-1751 (SENSITIVE_PERMISSIONS, installPermissionHandlers, console.warn-only denial at 1739); src/main/index.ts:1719-1730 (only type 'app' views can ever be granted)

**Recommended fix:** Route sensitive permission requests through the existing ConsentDialog/approval infrastructure (per-origin, remembered), or at minimum emit a notification:show ('Camera blocked for this panel') so the user learns why the feature failed; consider allowing fullscreen for browser panels by default.

### Renderer killed by the OS leaves a permanently blank panel — 'killed'/'abnormal-exit' reasons never trigger crash recovery

_Surface: `panels-infra-notifications`_

ViewManager only invokes crash callbacks for render-process-gone reasons 'crashed', 'oom', and 'launch-failed'. On Linux/macOS memory pressure the reason is commonly 'killed' (or 'abnormal-exit'); in that case no crash callback fires, PanelView's own renderProcessGone handler only console.warns, so handleViewCrashed/reload/showPanelErrorPage never run. The user is left staring at a blank white panel with no error page, no notification, and no automatic reload — the unresponsive-panel UI doesn't fire either because the process is gone, not busy.

**Evidence:** src/main/viewManager.ts:584-588 (reason filter), src/main/panelView.ts:488-490 (renderProcessGone console.warn only), src/main/index.ts:1473-1475 (crash wiring only via onViewCrashed)

**Recommended fix:** Treat every non-'clean-exit' render-process-gone reason as a crash (or at least 'killed' and 'abnormal-exit'), routing through handleViewCrashed so the reload/error-page policy applies.

### Outbox reordering (drag and the new arrow buttons) is purely cosmetic — delivery order never changes

_Surface: `chat-agent-interaction`_

Queued unread messages can be reordered by drag or by the Move up/down buttons added on this branch, but both paths only write a local orderOverride state used to re-sort the rendered list. The messages are already durably published to the channel; nothing retracts/re-publishes or transmits the new order, and the agent folds the log in original sequence. The user's explicit intent ('deliver B before A') is silently ignored, and a second panel on the same channel shows a different order. The a11y fix amplified the illusion by making the no-op control keyboard-reachable.

**Evidence:** workspace/packages/agentic-chat/components/Outbox.tsx:139-191 (orderOverride local state; handleDrop/handleMove only call setOrderOverride), 214-233 (render order only); no channel call anywhere in the reorder path; git diff 4c525dff added onMove arrows (OutboxItem.tsx:183-207)

**Recommended fix:** Either wire reorder to a real mechanism (retract + re-send in new order, or a delivery-order hint the agent honors) or remove the reorder affordances and keep the queue read-only with edit/cancel.

### Minimized approval pill still blocks the entire app behind an invisible Modal

_Surface: `mobile-app`_

The new 'not now' affordance minimizes the ApprovalSheet to a small 'Approval waiting · Review' pill, but the pill is rendered inside a transparent full-screen React Native <Modal>. RN Modals host their content in a separate native window (Android Dialog / iOS presented controller) that intercepts ALL touches; pointerEvents="box-none" only affects hit-testing inside the modal's own hierarchy, never pass-through to the app underneath. So while an approval is minimized the user can see the workspace but cannot tap, scroll, or navigate anything — the only working control is the pill. The whole point of deferring (keep using the app, come back later) is defeated; the user is still walled until they decide.

**Evidence:** workspace/apps/mobile/src/components/ApprovalSheet.tsx:371-392 (minimized state returns <Modal visible transparent> wrapping the pill), :1913-1918 (minimizedRoot absoluteFill inside the modal). Tests (ApprovalSheet.test.tsx:426) only assert deny is not sent, not that the app stays interactive.

**Recommended fix:** Render the minimized pill outside any Modal — as an absolutely-positioned overlay in MainScreen (like Toast, which correctly uses a plain View with pointerEvents="box-none", Toast.tsx:58) — and only mount the Modal when the sheet is expanded.

### Freshly-paired desktop never switches from the one-time pairing code to the refresh credential, so the first network blip after pairing kills the session

_Surface: `remote-webrtc-desktop`_

establishFreshPairSession dials with getShellToken: () => pairing.code and never swaps to the refresh credential after onPaired delivers it (the mobile client does swap — packages/mobile-webrtc/src/connect.ts:336-345 calls tokenProvider.setCredential). Session reopens re-invoke getShellToken on every (re)open, so after any pipe drop the reopen presents the already-consumed one-time code, the server rejects it terminally (webrtcSessionShim.ts:367-374, terminal: true), and the main session terminates permanently. The user who just paired sees the app go dead on the first Wi-Fi blip or sleep/wake; every RPC fails with 'Session is closed' until they quit and relaunch (the persisted credential works on the next launch). The transport itself reconnects fine, so the connection badge stays green while the app is dead.

**Evidence:** src/main/serverSession.ts:364 (getShellToken: () => pairing.code, never updated); src/main/serverSession.ts:367-369 (onPaired only records the credential for persistence); packages/rpc/src/transports/webrtcClient.ts:1731-1742 (terminal open rejection → terminate); packages/rpc/src/transports/pairedConnection.ts:257 (getToken re-invoked per reopen); contrast packages/mobile-webrtc/src/connect.ts:270,341-345 (mobile swaps to refresh token)

**Recommended fix:** In establishFreshPairSession, hold the issued credential in a mutable ref and make getShellToken return `refresh:<deviceId>:<refreshToken>` once onPaired fires (mirroring makeShellTokenProvider in mobile-webrtc), so reconnects authenticate with the durable credential.

### Terminal death of the main shell session leaves the badge green and the app silently broken

_Surface: `remote-webrtc-desktop`_

When the main shell session is terminally closed while the DTLS pipe stays up or reconnects (device revoked from another client, consumed pairing code on reopen, lease-style terminal close), SessionImpl.terminate removes the session but nothing notifies the shell: webrtcServerClient only forwards transport.onStatusChange, and transport.status() still reads 'connected'. The badge shows green 'Connected to <host>', getCurrent reports isActive: true, and every RPC fails — the user faces a dead app with a healthy-looking connection indicator and no re-pair/relaunch prompt. The launch-time terminal-credential recovery (index.ts:2092-2099) never runs mid-session.

**Evidence:** packages/rpc/src/transports/webrtcClient.ts:1801-1829 (terminate leaves transport untouched, no callback to the app layer); src/main/webrtcServerClient.ts:127,302-307 (only transport status forwarded; isConnected() = transport.status()); src/main/index.ts:250-255,2092-2099 (terminal credential handling is launch-only)

**Recommended fix:** Expose a main-session terminal-close callback from createPairedConnection/webrtcServerClient, and on it emit server-connection-changed status 'disconnected' with a reason, show a 're-pair or relaunch' prompt, and run the same isTerminalRemoteCredentialFailure recovery used at launch.

### Corrupt capability-grants.json crashes the server at startup with no recovery path

_Surface: `trust-permissions-credentials`_

CapabilityGrantStore.load() rethrows any non-ENOENT error, including JSON.parse SyntaxError from a corrupt or truncated capability-grants.json (SyntaxError has no .code, so the ENOENT check rethrows). The store is constructed unconditionally during server boot, so a single corrupt state file makes the whole app fail to start with an opaque exception and no guidance. The sibling stores handle this gracefully: UserlandApprovalGrantStore warns and starts empty, CredentialUseGrantStore warns and ignores the file — capabilityGrantStore is the inconsistent outlier.

**Evidence:** src/server/services/capabilityGrantStore.ts:142-153 (rethrow on non-ENOENT); src/server/index.ts:602 (constructed at boot); contrast src/server/services/userlandApprovalGrantStore.ts:159-164 and src/server/services/credentialUseGrantStore.ts:55-66 (warn + start empty)

**Recommended fix:** Match the sibling stores: on unreadable/corrupt grants file, log a warning (ideally surface a shell notification that saved permissions were reset) and start with an empty grant list, optionally renaming the bad file aside instead of crashing the server.

### Durable credential-use grants ('Trust repo/version' on credential approvals) cannot be individually revoked and are missing from the Permissions page

_Surface: `trust-permissions-credentials`_

When a user answers a credential approval with 'Trust repo' or a trust-version option, the grant is persisted in CredentialUseGrantStore, which exposes only list() and upsert() — there is no revoke API at any layer. The Credentials page renders these grants read-only (GrantRow has expand and focus-subject only, no revoke button), and the new Permissions page — whose subtitle is 'Lasting access you granted to apps and agents' — lists only capability and userland grants, so these lasting grants never appear there either. The only recourse is revoking the entire credential, which destroys the token upstream — wildly disproportionate to 'stop this one app from using it'. The Credentials page even cross-links to Permissions for 'lasting app or agent access', sending users to a page where these grants do not exist.

**Evidence:** src/server/services/credentialUseGrantStore.ts:10-13,24-39 (list/upsert only, no revoke); workspace/about/credentials/index.tsx:283-354 (GrantRow: no revoke control); src/server/services/permissionsService.ts:36-73 (list covers only capabilityGrants + userlandGrants); workspace/about/credentials/index.tsx:600-603 (cross-link claiming Permissions covers lasting access); src/server/services/credentialService.ts:4396-4403 (persistCredentialUseGrant upsert)

**Recommended fix:** Add a revoke(credentialId, grantKey) API to CredentialUseGrantStore, expose it through permissionsService (or credentials service), include credential-use grants in the Permissions list, and add a per-grant Revoke button to the Credentials page GrantRow.

### workspace.select with an unknown name silently creates and auto-trusts a brand-new workspace

_Surface: `workspace-projects-files`_

The server-side workspace.select handler never validates that the target workspace exists: touchWorkspace silently no-ops for unknown names and the relaunch event fires anyway. The shell relaunches with '--workspace <name> --workspace-create-if-missing', so a typo'd or hallucinated name (from eval/agent code calling workspace.switchTo, which the SKILL.md documents) creates a fresh workspace from the shipped template instead of erroring. Worse, because the workspace was just created, autoApproveStartupUnits flips true and the new workspace's template code (12 extensions, apps) starts without the trust gate. The user approves a 'Switch workspace?' prompt that never reveals the target does not exist, then lands in an unfamiliar empty workspace believing their data is gone.

**Evidence:** src/server/services/workspaceService.ts:585-599 (no hasWorkspace check before emitting relaunch); packages/shared/src/centralData.ts:144-160 (touchWorkspace silently returns for missing dir); src/main/startupMode.ts:146-153 (workspaceRelaunchArgs always appends WORKSPACE_CREATE_IF_MISSING_ARG) and 194-197 (autoApproveStartupUnits = created); packages/workspace/src/loader.ts:593-605 (init:true creates from template)

**Recommended fix:** In workspace.select, reject unknown names with a clear error ('Workspace "x" does not exist; use workspace.create first' plus close-match suggestion) before touching the catalog or emitting the relaunch event; drop --workspace-create-if-missing from workspaceRelaunchArgs (select should never create).

### Extension build/activation failures are invisible — extensions:error has no subscriber and there is no units status UI

_Surface: `workspace-projects-files`_

When an extension fails to build or activate at boot reconcile or after a source push, the registry is patched to status 'error' with lastError, and an 'extensions:error' event is emitted — but nothing in the desktop shell or Electron main subscribes to that event (only 'extensions:status' is bridged, and solely to retry host-target launches), and no panel or shell surface lists workspace units with status/health/lastError. **Evidence:** packages/extension-host/src/service.ts:1694-1703 (handleSourceRebuilt catch: registry.patch + emit extensions:error only); packages/unit-host/src/index.ts:729-733 (applyRuntimeDeclaration catch → markError, no notification); src/main/serverEventBridge.ts:223-225 (only extensions:status handled, and only for host-target retry); no grep hits for extensions:error subscribers outside extension-host; EXTENSIONS.md:750-787 (promised unified status surface/panel); only consumers of units.list are apps/remote-cli and shell/client.ts helper with no component using it for status

**Recommended fix:** Route extension error status changes through notificationService (like onCrashLimit already does) with a Reload action, and ship a minimal units status section (e.g. in about/ or settings) listing kind/status/health/lastError with restart buttons.
